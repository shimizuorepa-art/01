# ティアの会連動 お見送りアプリ 設計メモ

## 目的

当日にワンタップで会館へ繋ぐための準備アプリ。平常時に必要情報を登録しておき、亡くなった直後は「お見送りの依頼」ボタンだけで会館担当者へ正確な初動情報を渡す。

このアプリの主目的は、終活全般や葬儀後相談ではなく、当日の初動連絡を迷わず・漏れなく・負担なく進めること。

公開向けには、株式会社ティアおよびティアの会の正式許諾、会員システム/API仕様、ブランドガイドライン、個人情報保護・運用体制の確認が必要。

## 想定ユーザー

- ティアの会 会員本人
- 会員の家族・喪主予定者
- 会館担当者、コールセンター、搬送担当者
- 寺院手配・日程調整担当者

## 主要機能

### 会員機能

- デジタルティアカード表示
- 会員番号、会員種別、利用可能会館の確認
- 生き方応援ポイントの確認
- 登録住所、電話番号、代表連絡者の確認・変更
- 会員優待サービスへの導線

### お見送り事前準備

- 本人・家族・喪主予定者の基本情報
- 希望会館、搬送先、安置希望
- かかりつけ医、病院、施設、自宅などの所在地
- 宗教・宗派
- 菩提寺の有無、寺院連絡先
- 寺院側の対応可能日、不可日、友引対応方針
- 菩提寺が対応不可の場合の提携寺院候補
- 第一報を出したい親族・知人・会社関係者
- 第一報文面テンプレート
- 即時送信可 / 家族確認後 / 担当者確認後 の送信ルール
- 代表連絡者、搬送補足、安置希望、希望会館

### ワンタップ依頼

「お見送りの依頼」を押すと、次の情報を依頼ペイロードとして送信する。

- 会員ID
- 依頼者ID
- 代表連絡先
- 現在地または搬送元
- 希望会館・搬送先
- 宗派・菩提寺情報
- 寺院候補とスケジュール
- 第一報通知ルール
- 事前準備登録状況
- 受付日時、端末、位置情報許可状態

送信後、利用者画面は情報量を減らし、以下のみを表示する。

- 受付済みメッセージ
- 担当者へ電話
- 受付番号
- 現在の対応状況
- 担当者へ共有済みの登録内容

## UX方針

- 当日は判断を求めない。
- 文字は大きく、ボタンは44px以上にする。
- 警告色や強い通知音を避ける。
- 依頼後は通常の会員機能を隠し、電話・受付状況・共有内容に集中する。
- 知人への第一報は誤送信リスクが高いため、即時送信対象は事前に明確な同意がある連絡先だけに限定する。
- 家族代表や会館担当者への通知は即時、一般知人や会社関係者は確認後送信を標準にする。

## ティア導線研究からのUI原則

公開情報上、ティアは「お客様の状況を選ぶ」「電話で伺う内容を明示する」「ティアカードを提示するだけ」「公式LINEのマイページで会員情報へアクセスする」など、行動単位を短く見せる導線が強い。

本アプリでは、その思想をさらに進める。

- 入力を一度に出さず、Level 1からLevel 3と当日依頼に分ける。
- Level 1はワンタップ依頼に必要な最小情報だけにする。
- Level 2は会館連絡を確実にする情報にする。
- Level 3は初動後の確認を減らす任意情報にする。
- 当日は登録済み情報を会館へ送信し、担当者からの連絡を待つだけにする。
- 入力中は「あと何項目」ではなく「今はここまでで十分」を表示する。
- 登録済み情報は、当日画面では確認用に留め、再入力させない。
- 依頼後は、会員証・ポイント・優待など通常時の機能を隠し、受付状況と担当者電話だけに絞る。

### ビジュアル方針

- ティア公式導線に合わせ、緑・白・ゴールドを基調にする。
- 上部にTEARロックアップと24時間365日受付を配置し、電話への退避導線を常に残す。
- 会員証、ポイント、事前準備をカード化し、ティアの会/TEAR+の会員体験から自然に入れる。
- CTAは「申し込み」ではなく「会館へ繋ぐ」ための行動として見せる。
- 装飾よりも、登録済み・未確定・次の行動が一目で分かることを優先する。

## ティア側管理画面のUI原則

会館スタッフ側は、情報をたくさん見せることより「次に何をすればよいか」を最上段で判断できることを優先する。

- 最上段に次の行動を表示する。
- 搬送判断、安置判断、折り返し判断を分ける。
- 初動の必須確認と任意登録情報を混ぜない。
- 必須確認は、未確定だけが残るチェックリストにする。
- 任意情報は会館連絡の補足情報として別枠表示する。
- 第一報、寺院確認、担当者割当など、次アクションのボタンを常時見える位置に置く。
- スタッフが電話しながら見ても読めるよう、1項目1判断の粒度にする。

## 初動負荷削減の設計

ティア公式の「お急ぎの方へ」で電話時に伺う内容を、平常時の事前登録と当日の最小確認に分ける。

### 事前登録で埋める項目

- 故人様のお名前
- ご連絡者様のお名前・ご連絡先
- お迎えにあがる病院・施設名、部屋番号
- 搬送先
- ご自宅の住所・電話番号
- お迎えにあがる時間の候補

### 当日に確認する項目

- ご連絡者様への折り返し先
- 病院・施設側の退院準備とお迎え可能時刻
- 死亡診断書の受け取り状況
- 安置先の最終確認

ユーザー側はワンタップ後、受付済みメッセージ、担当者電話、受付状況、共有済み内容だけを表示する。ティア側は登録済み情報を先に確認し、未確定項目だけを電話で確認する。

この設計により、ユーザーは混乱した状態で同じ情報を何度も説明しなくて済み、ティア側は初回電話の聞き取り漏れ、転記、確認待ちを減らせる。

### 任意で登録できる詳しい情報

必須の初動6項目とは分けて、任意の補足情報を登録できるようにする。登録がなくてもワンタップ依頼はできるが、登録があるほど会館への初動連絡の精度が上がる。

- 優先する代表連絡者
- 搬送時の補足事項
- 安置希望
- 宗派、菩提寺、寺院連絡先の補足
- 第一報を出す親族・知人の範囲
- 第一報メール文面
- 希望会館、搬送先、駐車場、病棟入口などの補足メモ

管理画面側では、任意情報を初動の必須確認とは別枠で表示する。担当者はまず必須確認だけを進め、会館連絡・搬送判断に必要な時点で任意情報を参照する。

## 運用上の注意

- 死亡診断書、死亡届、火葬許可などの法的手続きは、アプリ上では「次に必要なこと」として案内し、実務は担当者確認を前提にする。
- 位置情報、家族情報、寺院情報、第一報連絡先はセンシティブ情報として扱う。
- ワンタップ依頼の取り消し、重複送信、いたずら送信対策が必要。
- 夜間受付、コールセンター、会館担当者、搬送担当者への通知ルーティングを運用設計に含める。
- API障害時は電話導線を最優先で表示する。

## プロトタイプ

静的プロトタイプは `tear-omimokuri-prototype/index.html`。

主な確認ポイント:

- 通常時は会員アプリとして使える
- 「お見送りの依頼」を押すと依頼後モードへ移行する
- 依頼後は最低限の情報だけ表示される
- 担当者電話、受付状況、共有済み登録内容を確認できる
